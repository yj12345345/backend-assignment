# Payment System

## 1. 프로젝트 개요

본 프로젝트는 회원 등급에 따른 할인과 결제 수단에 따른 추가 할인을 적용하는 결제 시스템을 구현하는 것을 목표로 합니다.  
실제 서비스 환경처럼 할인 정책이 자주 변경될 수 있다는 점을 고려하여, 유연한 구조와 이력 보존에 집중하여 설계했습니다.

---

## 2. 요구사항 정리

### 기본 기능 (main)
- 회원 등급: NORMAL, VIP, VVIP
- 할인 정책
  - VIP: 1000원 할인
  - VVIP: 10% 할인
- 결제 수단: CARD, POINT
- 결제 정보 저장

### 확장 기능 (feature)
- 할인 이력 저장
- 결제 수단 중복 할인 (POINT: 5%)
- 정책 변경 이후에도 과거 데이터 유지

---

## 3. 설계 과정 및 고민

### 3.1 할인 정책 분리

초기에는 단순 if문으로 처리할 수 있었지만, 정책이 계속 추가될 수 있다고 판단하여 다음과 같이 분리했습니다.

- GradeDiscountPolicy
- PaymentDiscountPolicy

이를 통해 정책 변경 시 기존 코드 수정 없이 확장만으로 대응할 수 있도록 설계했습니다.

---

### 3.2 할인 이력 저장 방식

정책이 변경되어도 과거 결제 데이터는 유지되어야 하기 때문에, 단순 계산 결과만 저장하는 방식은 적절하지 않다고 판단했습니다.

그래서 Payment 내부에 할인 결과를 별도로 저장하도록 구성했습니다.

- Payment
  - discountResults (List<DiscountResult>)

각 할인 결과에는 다음 정보가 포함됩니다:
- 정책 이름
- 할인 금액
- 할인율
- 적용된 회원 등급

이 구조를 통해 과거 결제 데이터 재현이 가능하도록 했습니다.

---

### 3.3 DiscountResult를 값 객체로 설계

DiscountResult는 독립적인 엔티티가 아니라 결제에 종속된 데이터라고 판단했습니다.

따라서 @ElementCollection을 사용하여 다음과 같이 처리했습니다:

- Payment에 종속된 값 객체
- 별도의 생명주기 없음
- 단순 이력 저장 목적

---

### 3.4 중복 할인 처리 순서

중복 할인에서 가장 중요하게 고려한 부분은 할인 적용 순서였습니다.

현재 적용 순서는 다음과 같습니다:

1. 회원 등급 할인 적용
2. 할인된 금액 기준으로 결제 수단 할인 적용

이 순서를 유지함으로써 정책 간 충돌을 방지하고 일관된 결과를 보장했습니다.
